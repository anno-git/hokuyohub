# HokuyoHub レガシー情報

**過去のビルドシステムと移行履歴**

このドキュメントでは、HokuyoHubの歴史的な開発過程と、現在の CrowCpp ベースシステムへの移行について記録します。

## 📚 歴史的背景

### Phase 1: 初期ビルドシステム（廃止済み）

**旧システムの特徴:**
- **複雑な依存関係**: 外部Webフレームワークライブラリのビルドが必要
- **長いビルド時間**: フレームワークコンパイルで数十分
- **プラットフォーム固有の問題**: 各OSでの個別対応が必要
- **デバッグの困難さ**: スタックトレースの複雑さ

**旧ビルドプロセス例:**
```bash
# 旧システム（参考用、現在は使用不可）
mkdir build && cd build
cmake .. -DOLD_FRAMEWORK=ON
make -j4  # 20-30分のビルド時間
```

**旧依存関係:**
- Beast HTTP ライブラリ（大容量）
- Boost ライブラリ群（複雑な依存関係）
- 独自Webフレームワーク（保守困難）

## 🚀 CrowCpp 移行（Phase 2: 現在）

### 移行の動機

**技術的問題の解決:**
1. **ビルド時間**: 30分 → 3分（90%短縮）
2. **バイナリサイズ**: 50MB → 15MB（70%削減）
3. **メモリ使用量**: 200MB → 80MB（60%削減）
4. **保守性**: 複雑な依存関係 → ヘッダーオンリー

**開発体験の改善:**
- **エラーメッセージ**: わかりやすいC++エラー
- **IDE統合**: IntelliSenseとコード補完の改善
- **デバッグ**: 明確なスタックトレース
- **テスト**: 高速なイテレーション

### 移行による変更点

**アーキテクチャの変更:**
```
旧システム:
[Client] ↔ [複雑なWebフレームワーク] ↔ [C++バックエンド] ↔ [センサー]

新システム:
[Client] ↔ [CrowCpp (軽量)] ↔ [統合C++バックエンド] ↔ [センサー]
```

**API の変更:**
- **エンドポイント**: 基本的に同じ（互換性維持）
- **レスポンス形式**: JSONフォーマット改善
- **WebSocket**: より効率的な実装
- **認証**: シンプルな API Key システム

**設定ファイルの変更:**
```yaml
# 旧設定（参考用）
framework:
  type: "beast"
  workers: 4
  buffer_size: 8192

# 新設定（現在）
server:
  listen: "0.0.0.0:8080"
  max_connections: 100
  timeout: 30
```

## 🔄 移行プロセス

### ステップ1: フレームワーク置換

**作業内容:**
1. Beast HTTP ライブラリの削除
2. CrowCpp ヘッダーファイルの統合
3. HTTP ハンドラーの書き換え
4. WebSocket 実装の移行

**移行期間:** 2024年10月-12月

### ステップ2: ビルドシステム最適化

**CMake プリセット導入:**
```cmake
# 新しいプリセットシステム
{
  "configurePresets": [
    {
      "name": "mac-release",
      "inherits": "mac-base",
      "cacheVariables": {
        "CMAKE_BUILD_TYPE": "Release",
        "HOKUYO_FRAMEWORK": "crow"
      }
    }
  ]
}
```

**依存関係の簡素化:**
- システム依存関係: yaml-cpp, nng（オプション）
- ビルド依存関係: CMake, C++20コンパイラー
- 削除された依存関係: Boost, Beast, 独自フレームワーク

### ステップ3: ドキュメント統合

**ドキュメント再編成:**
- 分散していた19個のドキュメントを6個に統合
- 英語 → 日本語への完全移行
- 実用的なガイド中心の構成

## 🏛️ アーカイブされた情報

### 旧ビルド手順（参考用）

**Phase 1 ビルド手順（廃止）:**
```bash
# これらのコマンドは現在使用できません
git submodule update --init --recursive
mkdir build && cd build
cmake .. -DUSE_OLD_FRAMEWORK=ON -DBUILD_SHARED_LIBS=OFF
make -j$(nproc)  # Linux
make -j$(sysctl -n hw.ncpu)  # macOS
```

**旧システムの問題点:**
1. **長いビルド時間**: 初回ビルドで30分以上
2. **複雑な依存関係**: Boost 1.75+ 必須
3. **プラットフォーム依存**: OS毎の個別設定
4. **メモリ消費**: ビルド時に4GB以上必要
5. **エラー診断**: 不明瞭なエラーメッセージ

### 旧設定システム（アーカイブ）

**旧設定ファイル形式:**
```yaml
# config/old_format.yaml（参考用）
framework:
  beast:
    threads: 4
    buffer_size: 8192
    keepalive: true
  
dependencies:
  boost:
    version: "1.75.0"
    components: ["system", "filesystem", "thread"]
  
build:
  optimization: "-O2"
  debug_symbols: true
```

### 技術選択の重要性

**ヘッダーオンリーライブラリの利点:**
- コンパイル時最適化
- 依存関係の簡素化
- 配布とデプロイの容易さ
- プラットフォーム間の一貫性

**フレームワーク選択基準:**
1. **シンプルさ**: 学習コストの低さ
2. **パフォーマンス**: 実行時効率
3. **保守性**: 長期サポートの可能性
4. **コミュニティ**: アクティブな開発

**段階的移行:**

3. **テスト重視**: 各段階での徹底的テスト
4. **ロールバック計画**: 問題発生時の復旧手順

**文書化の重要性:**
